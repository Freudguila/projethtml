NOM

giteveryday - Un ensemble minimal de commandes utiles pour Git au quotidien

SYNOPSIS

Git au quotidien avec environ 20 commandes

DESCRIPTION

Les utilisateurs de Git peuvent être globalement regroupés en quatre catégories afin de décrire ici un petit ensemble de commandes utiles pour Git au quotidien.

  • Les commandes de développeur individuel (autonomes) sont essentielles pour quiconque s'engage, même pour quelqu'un qui travaille seul.

  • Si vous travaillez avec d’autres personnes, vous aurez également besoin des commandes répertoriées dans la section Développeur individuel (Participant) .

  • Les personnes qui jouent le rôle d’ intégrateur doivent apprendre quelques commandes supplémentaires en plus de celles ci-dessus.

  • Les commandes d'administration du référentiel sont destinées aux administrateurs système responsables de la gestion et de l'alimentation des référentiels Git.

Développeur individuel (autonome)

Un développeur individuel autonome n'échange pas de correctifs avec d'autres personnes et travaille seul dans un seul référentiel, à l'aide des commandes suivantes.

  • git-init[1] pour créer un nouveau référentiel.

  • git-log[1] pour voir ce qui s'est passé.

  • git-switch[1] et git-branch[1] pour changer de branche.

  • git-add[1] pour gérer le fichier d'index.

  • git-diff[1] et git-status[1] pour voir ce que vous êtes en train de faire.

  • git-commit[1] pour faire avancer la branche actuelle.

  • git-restore[1] pour annuler les modifications.

  • git-merge[1] pour fusionner entre les branches locales.

  • git-rebase[1] pour maintenir les branches thématiques.

  • git-tag[1] pour marquer un point connu.

Exemples

Utilisez une archive tar comme point de départ pour un nouveau référentiel.
$ tar zxf frotz.tar.gz
$ cd frotz
$ git init
$ git add . (1)
$ git commit -m "importation de l'arborescence source de frotz."
$ git tag v2.43 (2)

  1. ajouter tout ce qui se trouve sous le répertoire actuel.

  2. créer une balise légère et non annotée.

Créez une branche thématique et développez-la.
$ git switch -c alsa-audio (1)
$ éditer/compiler/tester
$ git restore curses/ux_audio_oss.c (2) 
$ git add curses/ux_audio_alsa.c (3)
$ éditer/compiler/tester
$ git diff HEAD (4) 
$ git commit -a -s (5)
$ éditer/compiler/tester
$ git diff HEAD^ (6) 
$ git commit -a --amend (7) 
$ git switch master (8) 
$ git merge alsa-audio (9) 
$ git log --since='il y a 3 jours' (10) 
$ git log v2.43.. curses/ (11)

  1. créer une nouvelle branche de sujet.

  2. annulez vos modifications bâclées dans curses/ux_audio_oss.c.

  3. vous devez indiquer à Git si vous avez ajouté un nouveau fichier ; la suppression et la modification seront détectées si vous le faites git commit -aplus tard.

  4. pour voir quels changements vous apportez.

  5. engagez tout, comme vous l'avez testé, avec votre signature.

  6. regardez tous vos changements, y compris le commit précédent.

  7. modifiez le commit précédent, en ajoutant toutes vos nouvelles modifications, en utilisant votre message d'origine.

  8. passer à la branche principale.

  9. fusionner une branche de sujet dans votre branche principale.

  10. examiner les journaux de validation ; d'autres formes de limitation de sortie peuvent être combinées et inclure -10(pour afficher jusqu'à 10 validations) --until=2005-12-10, etc.

  11. afficher uniquement les modifications qui touchent ce qui se trouve dans curses/ le répertoire, depuis v2.43la balise.

Développeur individuel (Participant)

Un développeur travaillant en tant que participant à un projet de groupe doit apprendre à communiquer avec les autres et utilise ces commandes en plus de celles dont un développeur autonome a besoin.

  • git-clone[1] depuis l'amont pour amorcer votre référentiel local.

  • git-pull[1] et git-fetch[1] depuis « origin » pour rester à jour avec l'amont.

  • git-push[1] vers un référentiel partagé, si vous adoptez le flux de travail du référentiel partagé de style CVS.

  • git-format-patch[1] pour préparer la soumission par e-mail, si vous adoptez le flux de travail du forum public de style noyau Linux.

  • git-send-email[1] pour envoyer votre soumission par e-mail sans corruption par votre MUA.

  • git-request-pull[1] pour créer un résumé des modifications que votre amont doit extraire.

Exemples

Clonez l'amont et travaillez dessus. Transmettez les modifications à l'amont.
$ git clone git://git.kernel.org/pub/scm/.../torvalds/linux-2.6 my2.6
$ cd my2.6
$ git switch -c mine master (1) 
$ edit/compile/test; git commit -a -s (2) 
$ git format-patch master (3) 
$ git send-email --to="personne <email@exemple.com>" 00*.patch (4) 
$ git switch master (5) 
$ git pull (6) 
$ git log -p ORIG_HEAD.. arch/i386 include/asm-i386 (7) 
$ git ls-remote --heads http://git.kernel.org/.../jgarzik/libata-dev.git (8) 
$ git pull git://git.kernel.org/pub/.../jgarzik/libata-dev.git ALL (9) 
$ git reset --hard ORIG_HEAD (10) 
$ git gc (11)

  1. extraire une nouvelle branche minede master.

  2. répéter si nécessaire.

  3. extraire les correctifs de votre branche, par rapport au maître,

  4. et envoyez-leur un e-mail.

  5. retour à master, prêt à voir les nouveautés

  6. git pullrécupère originpar défaut et fusionne dans la branche actuelle.

  7. immédiatement après avoir tiré, regardez les changements effectués en amont depuis la dernière fois que nous avons vérifié, uniquement dans la zone qui nous intéresse.

  8. vérifiez les noms des branches dans un référentiel externe (si non connus).

  9. récupérer à partir d'une branche spécifique ALLd'un référentiel spécifique et le fusionner.

  10. inverser la traction.

  11. les déchets collectent les objets restants de l'extraction inversée.

Pousser vers un autre référentiel.
satellite$ git clone mothership:frotz frotz (1)
satellite$ cd frotz
satellite$ git config --get-regexp '^(remote|branch)\.' (2)
remote.origin.url vaisseau-mère :frotz
remote.origin.fetch refs/heads/*:refs/remotes/origin/*
branche.maître.origine distante
branche.master.merge refs/heads/master
satellite$ git config remote.origin.push \
	   +refs/heads/*:refs/remotes/satellite/* (3)
satellite$ éditer/compiler/tester/valider
satellite$ git push origin (4)

vaisseau-mère$ cd frotz
vaisseau-mère$ git switch master
vaisseau-mère$ git merge satellite/master (5)

  1. La machine mère possède un référentiel frotz sous votre répertoire personnel ; clonez-le pour démarrer un référentiel sur la machine satellite.

  2. Le clone définit ces variables de configuration par défaut. Il git pullrécupère et stocke les branches de la machine mère dans remotes/origin/*des branches locales de suivi à distance.

  3. organiser git pushle déplacement de toutes les branches locales vers leur branche correspondante de la machine mère.

  4. La commande push stockera tout notre travail sur remotes/satellite/* des branches de suivi à distance de la machine mère. Vous pouvez utiliser cette méthode comme méthode de secours. De même, vous pouvez faire comme si la machine mère avait récupéré les données (utile lorsque l'accès est unilatéral).

  5. sur la machine mère, fusionnez le travail effectué sur la machine satellite dans la branche principale.

Branche à partir d'une balise spécifique.
$ git switch -c private2.6.14 v2.6.14 (1)
$ edit/compile/test; git commit -a
$ git checkout master
$ git cherry-pick v2.6.14..private2.6.14 (2)

  1. créer une branche privée basée sur une balise bien connue (mais quelque peu en retard).

  2. Transférer toutes les modifications d' private2.6.14une branche à masterl'autre sans fusion formelle. Ou écriture manuscrite
    git format-patch -k -m --stdout v2.6.14..private2.6.14 | git am -3 -k

Un autre mécanisme de soumission des participants consiste à utiliser les git request-pullmécanismes de demande d'extraction (par exemple, tels qu'utilisés sur GitHub (www.github.com) pour informer votre amont de votre contribution.

Intégrateur

Une personne assez centrale agissant comme intégrateur dans un projet de groupe reçoit les modifications apportées par d'autres, les examine et les intègre et publie le résultat pour que d'autres puissent l'utiliser, en utilisant ces commandes en plus de celles dont les participants ont besoin.

Cette section peut également être utilisée par ceux qui répondent à git request-pullune demande d'extraction sur GitHub (www.github.com) afin d'intégrer le travail d'autres personnes à leur historique. Un sous-lieutenant de zone pour un dépôt agira à la fois comme participant et comme intégrateur.

  • git-am[1] pour appliquer les correctifs envoyés par courrier électronique par vos contributeurs.

  • git-pull[1] pour fusionner à partir de vos lieutenants de confiance.

  • git-format-patch[1] pour préparer et envoyer des alternatives suggérées aux contributeurs.

  • git-revert[1] pour annuler les commits ratés.

  • git-push[1] to publish the bleeding edge.

Examples

A typical integrator’s Git day. 
$ git status (1)
$ git branch --no-merged master (2)
$ mailx (3)
& s 2 3 4 5 ./+to-apply
& s 7 8 ./+hold-linus
& q
$ git switch -c topic/one master
$ git am -3 -i -s ./+to-apply (4)
$ compile/test
$ git switch -c hold/linus && git am -3 -i -s ./+hold-linus (5)
$ git switch topic/one && git rebase master (6)
$ git switch -C seen next (7)
$ git merge topic/one topic/two && git merge hold/linus (8)
$ git switch maint
$ git cherry-pick master~4 (9)
$ compile/test
$ git tag -s -m "GIT 0.99.9x" v0.99.9x (10)
$ git fetch ko && for branch in master maint next seen (11)
    do
	git show-branch ko/$branch $branch (12)
    done
$ git push --follow-tags ko (13)

  1. see what you were in the middle of doing, if anything.

  2. see which branches haven’t been merged into master yet. Likewise for any other integration branches e.g. maint, next and seen.

  3. read mails, save ones that are applicable, and save others that are not quite ready (other mail readers are available).

  4. apply them, interactively, with your sign-offs.

  5. create topic branch as needed and apply, again with sign-offs.

  6. rebase internal topic branch that has not been merged to the master or exposed as a part of a stable branch.

  7. restart seen every time from the next.

  8. and bundle topic branches still cooking.

  9. backport a critical fix.

  10. create a signed tag.

  11. make sure master was not accidentally rewound beyond that already pushed out.

  12. In the output from git show-branch, master should have everything ko/master has, and next should have everything ko/next has, etc.

  13. push out the bleeding edge, together with new tags that point into the pushed history.

In this example, the ko shorthand points at the Git maintainer’s repository at kernel.org, and looks like this:

(in .git/config)
[remote "ko"]
	url = kernel.org:/pub/scm/git/git.git
	fetch = refs/heads/*:refs/remotes/ko/*
	push = refs/heads/master
	push = refs/heads/next
	push = +refs/heads/seen
	push = refs/heads/maint

Repository Administration

A repository administrator uses the following tools to set up and maintain access to the repository by developers.

  • git-daemon[1] to allow anonymous download from repository.

  • git-shell[1] can be used as a restricted login shell for shared central repository users.

  • git-http-backend[1] provides a server side implementation of Git-over-HTTP ("Smart http") allowing both fetch and push services.

  • gitweb[1] provides a web front-end to Git repositories, which can be set-up using the git-instaweb[1] script.

update hook howto has a good example of managing a shared central repository.

In addition there are a number of other widely deployed hosting, browsing and reviewing solutions such as:

  • gitolite, gerrit code review, cgit and others.

Examples

We assume the following in /etc/services 
$ grep 9418 /etc/services
git		9418/tcp		# Git Version Control System
Run git-daemon to serve /pub/scm from inetd. 
$ grep git /etc/inetd.conf
git	stream	tcp	nowait	nobody \
  /usr/bin/git-daemon git-daemon --inetd --export-all /pub/scm

The actual configuration line should be on one line.

Run git-daemon to serve /pub/scm from xinetd. 
$ cat /etc/xinetd.d/git-daemon
# default: off
# description: The Git server offers access to Git repositories
service git
{
	disable = no
	type            = UNLISTED
	port            = 9418
	socket_type     = stream
	wait            = no
	user            = nobody
	server          = /usr/bin/git-daemon
	server_args     = --inetd --export-all --base-path=/pub/scm
	log_on_failure  += USERID
}

Check your xinetd(8) documentation and setup, this is from a Fedora system. Others might be different.

Give push/pull only access to developers using git-over-ssh.

e.g. those using: $ git push/pull ssh://host.xz/pub/scm/project

$ grep git /etc/passwd (1)
alice:x:1000:1000::/home/alice:/usr/bin/git-shell
bob:x:1001:1001::/home/bob:/usr/bin/git-shell
cindy:x:1002:1002::/home/cindy:/usr/bin/git-shell
david:x:1003:1003::/home/david:/usr/bin/git-shell
$ grep git /etc/shells (2)
/usr/bin/git-shell

  1. log-in shell is set to /usr/bin/git-shell, which does not allow anything but git push and git pull. The users require ssh access to the machine.

  2. in many distributions /etc/shells needs to list what is used as the login shell.
